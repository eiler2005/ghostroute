# Channel M Environment Inventory

Channel M is the service-only MAX egress lane for `maxtg_bridge`. It is not a
client channel and not Channel A/B/C failover.

This document owns the router_configuration view of Channel M. The companion
VPS host-layer view is documented in
[`praefectus-ai/docs/vps-runtime-map.md`](https://github.com/eiler2005/praefectus-ai/blob/main/docs/vps-runtime-map.md).

## Active Runtime Shape

```text
maxtg_bridge container on VPS
  -> HTTP CONNECT to Channel M proxy host
  -> compose extra_hosts maps proxy host to VPS docker bridge gateway
  -> VPS docker bridge:<channel-m-reverse-listen-port>
  -> OpenSSH remote-forward session opened by the home router
  -> router loopback:<channel-m-reverse-ingress-port>
  -> sing-box inbound channel-m-maxtg-reverse-egress
  -> direct-out / home WAN Russian IP
  -> MAX API/CDN
```

Telegram traffic is intentionally outside this path:

```text
maxtg_bridge container on VPS
  -> direct HTTPS
  -> Telegram Bot API
```

## Port and Exposure Model

| Port variable | Bind scope | Exposure | Notes |
|---|---|---|---|
| `channel_m_maxtg_reverse_listen_port` | VPS docker bridge | bridge container only | Active reverse Channel M listener; not public UFW/cloud-firewall exposure |
| `channel_m_maxtg_reverse_ingress_port` | router loopback | SSH remote-forward target only | Plain HTTP CONNECT is safe here because it is inside SSH |
| `channel_m_maxtg_ingress_port` | router interfaces | optional direct public lane target | Kept isolated for controlled direct-lane experiments |
| `channel_m_maxtg_public_port` | home WAN redirect | optional direct public lane | Not required by active reverse Channel M |

## Redeploy Invariants

- `20-stealth-router.yml` may refresh sing-box and firewall state, but must keep
  `channel-m-maxtg-reverse-egress -> direct-out`.
- `23-channel-m-reverse.yml` is the idempotent owner of the router SSH
  remote-forward, VPS `permitlisten`, and VPS docker bridge firewall
  persistence.
- `99-verify.yml` must confirm the VPS docker bridge listener, firewall
  persistence, supervisor-owned router cron recovery, router boot recovery
  hook, router loopback listener and direct-only sing-box routing.
- No Channel M rule may add `rule_set`, `reality-out`, DNS policy, LAN/Wi-Fi
  ownership or Channel A/B/C failover behavior.
- No public home inbound port is required for the active reverse lane.

## Router Reboot Recovery

Channel M carries the messenger egress path, so it is recovery-critical. The
runtime supervisor re-registers the `ChannelMReverse` cron on every boot and
only invokes `/jffs/scripts/channel-m-reverse-tunnel.sh` after the local
sing-box reverse ingress is listening. A tunnel drop or stale reverse-forward is
therefore repaired by the Channel M script itself, without changing Channel
A/B/C/D routing ownership or restarting unrelated services.

There are two supported recovery modes:

1. Router boot/recover mode: `/jffs/scripts/services-start` launches
   `/jffs/scripts/ghostroute-runtime-supervisor.sh boot`, which restores the
   shared router runtime in dependency order and registers `ChannelMReverse`.
2. VPS/app on-demand mode: `maxtg_bridge` or a VPS-side watchdog may request
   only Channel M recovery by running the router command
   `/jffs/scripts/ghostroute-runtime-supervisor.sh channel-m-recover` through a
   restricted SSH profile. This command must not be used as a generic router
   restart hook: it checks the Channel M local ingress, restarts sing-box only if
   that ingress is missing, and then delegates stale tunnel replacement to
   `/jffs/scripts/channel-m-reverse-tunnel.sh`.

The read-only companion command is:

```sh
/jffs/scripts/ghostroute-runtime-supervisor.sh channel-m-status
```

It reports local ingress, cron and reverse tunnel state for app-side backoff
logic. The VPS/app profile should rate-limit recovery requests and keep the SSH
target/key in runtime secrets, not in tracked docs or application config.

**Manual recovery via control machine** — if the reverse tunnel is stale (router
reports `channel_m_reverse_tunnel=running` but the VPS watchdog detects no HTTP
CONNECT response), kill the stale sshd process on the VPS that holds the listen
port, then trigger channel-m-recover on the router via the control-machine SSH key
(`ghostroute_router_remote_private_key` in vault) to the router's WAN IP
(`ghostroute_router_remote_host:ghostroute_router_remote_port`):

```sh
# Check stale sshd pid on VPS
sudo ss -ltnp "( sport = :<channel-m-reverse-listen-port> )"
sudo kill <sshd-pid>

# Force tunnel re-establishment from control machine
ssh -i <router-remote-key> admin@<router-wan-ip> \
  '/jffs/scripts/ghostroute-runtime-supervisor.sh channel-m-recover'
```

The router-side script is deliberately stricter than a process-presence check.
Each cron or boot-delayed run takes a lock with `pid` and `started_at`
metadata, waits for VPS SSH reachability, checks
`127.0.0.1:<channel-m-reverse-ingress-port>`, verifies the remote VPS listener
and verifies that HTTP CONNECT through it reaches the router proxy, kills stale
reverse SSH sessions when validation fails, then starts a fresh `ssh -R`
tunnel. If an earlier watchdog run left a stale lock, the next run clears locks
older than 5 minutes before retrying. If local sing-box ingress is down after
boot, it attempts one sing-box restart before giving up to the next cron cycle.

The VPS-side listener watchdog covers the other half of router reboot recovery:
if an orphan `sshd` process keeps the VPS listener bound but the listener no
longer forwards to the router proxy, the watchdog kills that stale listener so
the next router cron cycle can create a fresh `ssh -R`.

This keeps the active lane fail-closed: Channel M recovers through the home
router path only. There is still no automatic fallback to `hetzner_direct`.
